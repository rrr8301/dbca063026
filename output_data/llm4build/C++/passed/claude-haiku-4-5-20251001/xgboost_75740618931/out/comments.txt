  [Root cause classification]
  - The `Build CPU (sanitizer)` CI job compiles XGBoost with AddressSanitizer (`-fsanitize=address`). ASan works by shadowing the entire virtual address space: it reserves a large contiguous region of memory (shadow memory) to track every byte allocation. In Docker containers with memory or virtual address space restrictions (`ulimit -v`), ASan cannot reserve its shadow memory ranges ("failed to allocate 0xff000 bytes", "ReserveShadowMemoryRange failed"). The test binary (`testxgboost`) aborts before running any tests, which is why the LLM's GTest count shows 389 tests "run" with 0 passed — the parser reads the test binary's registered tests count from the preamble but the binary crashes before executing them.

  - The original CI ran on a GitHub Actions Linux runner with full virtual address space (no ulimit restriction), allowing ASan to map its shadow memory (typically 16+ TB of virtual address space for x86_64). The 436 fewer tests in the LLM (389 vs 825) reflect which GTest test cases were registered before the crash, not executed. The `xgboost_66714523910` failed-category counterpart has the same dependency on pre-built artifacts (different CI job). This sanitizer memory restriction is the same class of problem as seen in other ASan-heavy projects in Docker: the container's virtual address space limit prevents the shadow memory reservation.
