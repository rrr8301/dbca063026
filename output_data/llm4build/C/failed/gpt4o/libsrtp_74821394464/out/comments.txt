  [Root cause classification]
  - The LLM run.sh clones and builds mbedTLS 4.0.0 (the latest release), but libsrtp's crypto layer uses the PSA Cryptography API (psa/crypto.h) which underwent a significant restructuring between mbedTLS 2.x/3.x and 4.0. In mbedTLS 4.0, the PSA Crypto API was moved to a separate TF-PSA-Crypto submodule and the include path changed, so the compilation of libsrtp's AES-GCM module fails with "fatal error: psa/crypto.h: No such file or directory".
  - Because the compilation fails, 9 of the 11 test binaries (datatypes_driver, cipher_driver, kernel_driver, rdbx_driver, replay_driver, roc_driver, srtp_driver, srtp_driver_not_in_place_io, test_srtp) are never built, so CTest reports them as "Not Run". The original CI uses a stable mbedTLS version (likely 2.x or 3.x via ubuntu-24.04 apt packages or an older git tag) where psa/crypto.h is available at the expected path.
  - This is an mbedTLS API breaking change issue: the LLM incorrectly assumed that the latest mbedTLS release would be compatible with libsrtp's crypto layer, but mbedTLS 4.0's PSA Crypto reorganization introduced headers that libsrtp's build system does not locate, causing a complete build failure of the cryptographic test executables.
