  [Root cause classification]
  Python version mismatch (3.14.5 vs 3.13.12) causes different test parametrization
  and 3 additional failures

  The original CI sets `UV_PYTHON=3.13` which causes uv to download and use
  cpython-3.13.12. The LLM Docker uses ubuntu:24.04 which provides Python 3.14.5 as
  the system Python. This major Python version difference (3.13 vs 3.14) has two effects:

  (1) 10 more tests collected in LLM (12246 vs 12236)
    pydantic's test suite has version-conditional test parametrization. Some tests are
    parametrized based on Python version capabilities (e.g., new Python 3.14 type system
    features, new syntax, or changed built-in behaviors). Python 3.14 enables additional
    test variants that don't exist for Python 3.13, adding 10 tests to the collection.
    Conversely, 10 fewer skips (455 vs 465) confirms that Python 3.14's expanded feature
    set allows some previously-skipped tests to now run.

  (2) 3 more failures in LLM (846 vs 843)
    The 3 additional failures are pydantic validation tests that produce different error
    messages or behaviors on Python 3.14 vs 3.13. Python 3.14 introduced changes to type
    error formatting, exception chaining, and decorator handling. When pydantic-core's Rust
    extension raises validation errors, the error message format can differ between Python
    versions if pydantic's expected error strings are pinned to Python 3.13 format.

  (3) CI job name "debug build" refers to Rust build profile, not Python
    The CI job is labeled "Test with debug build (3.13)" — "debug" refers to compiling
    the pydantic-core Rust extension with the Cargo `debug` profile ([unoptimized + debuginfo])
    rather than the `release` profile. Both CI and LLM Docker compile pydantic-core with the
    dev/debug Rust profile. The Python itself is a standard (non-debug) build in both cases.
    However, the CI has a Cargo cache restored for the debug build, while the LLM builds
    pydantic-core from scratch (taking ~10s in the Dockerfile).
