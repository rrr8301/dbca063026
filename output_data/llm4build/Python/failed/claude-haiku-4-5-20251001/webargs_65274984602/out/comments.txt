  setuptools==82.0.0 removed pkg_resources module → Pyramid fails to import → all pyramid
  tests ERROR at setup. LLM accidentally uses setuptools==81.0.0 where pkg_resources still
  works (with deprecation warning) → all pyramid tests pass

  The critical difference is the `setuptools` version used in the tox virtualenv:

  (1) Original CI: setuptools==82.0.0 — pkg_resources completely removed
    setuptools 82.0.0 executed the long-announced removal of `pkg_resources` as a standalone
    importable module (the deprecation warning stated "slated for removal as early as
    2025-11-30"). With setuptools==82.0.0, `import pkg_resources` raises:
      `ModuleNotFoundError: No module named 'pkg_resources'`
    pyramid 2.0.2's `pyramid/asset.py` still imports `pkg_resources` at line 2:
      `from pyramid.asset import resolve_asset_spec` triggers this import.
    Since this module-level import fails at test SETUP time, all 43 `TestPyramidParser` tests
    error before running a single assertion.

  (2) LLM Docker: setuptools==81.0.0 — pkg_resources deprecated but still importable
    The LLM Docker uses ubuntu:22.04 as base, and when tox creates the virtualenv, it seeds it
    with setuptools==81.0.0 (from the virtualenv seeder cache or installed pip). With
    setuptools==81.0.0, `pkg_resources` is still importable but emits a UserWarning:
      `UserWarning: pkg_resources is deprecated as an API... slated for removal as early as
       2025-11-30. Refrain from using this package or pin to Setuptools<81.`
      `import pkg_resources`
    The LLM also resolves to a newer pyramid version (2.1 vs 2.0.2), but the root difference
    is the setuptools version — even pyramid 2.0.2 works with setuptools 81 (with warning).
    All 43 pyramid tests that previously errored now run and pass.

  (3) This is a "flipped" reproducibility case
    The original CI FAILED because of a recent setuptools breakage (82.0.0 removing
    pkg_resources), not because of any bug in the webargs codebase itself. The LLM Docker
    accidentally avoids this failure by using a slightly older setuptools. From a build
    reproducibility perspective, the LLM produces BETTER results (0 failures) than the
    original CI (43 failures), but this is coincidental — it uses an earlier dependency
    version that predates the breaking change.
