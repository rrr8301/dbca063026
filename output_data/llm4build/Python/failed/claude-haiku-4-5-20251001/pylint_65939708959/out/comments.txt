  [Root cause classification]
  Python minor version bump (3.14.3 → 3.14.5) plus pytest/python-discovery version differences
  cause 85 more test skips and 1 fewer test collected

  The original CI runs Python 3.14.3 (from pyenv/hostedtoolcache) with pytest-9.0.2 and
  python-discovery-1.1.0. The LLM Docker installs the latest from pip, getting Python 3.14.5
  with pytest-9.0.3 and python-discovery-1.3.1.

  (1) 85 more skips in LLM (299 vs 214)
    pylint's test suite has many tests conditional on Python version behavior — specifically
    tests that check pylint's output for specific code patterns on specific Python versions.
    When Python 3.14.5 introduces new warnings, deprecations, or AST parsing changes compared
    to 3.14.3, tests that previously produced specific expected output on 3.14.3 now skip on
    3.14.5 because the expected output file doesn't match the current Python version. pylint
    uses `.txt` expected output files tied to Python version ranges; tests are skipped when
    no matching expected output file exists for the running Python version.

    Additionally, `python-discovery` (1.3.1 vs 1.1.0) can find a different set of Python
    interpreters on the system. Some pylint tests are parametrized based on which Python
    versions are discoverable — if python-discovery 1.3.1 discovers additional or fewer
    interpreters on the ubuntu:24.04 image vs the GitHub Actions runner, the parametrized
    test variants change.

  (2) 1 fewer test collected in LLM (2297 vs 2298)
    The single missing test is likely a parametrized test variant whose test ID changes
    between Python 3.14.3 and 3.14.5, or a test that is conditionally included based on
    available Python version ranges. With python-discovery 1.3.1 finding a different Python
    version set on ubuntu:24.04 vs ubuntu-latest runner, one parametrized version of a test
    may not be generated.

  (3) Pre-existing failure (same in both): test_do_not_import_files_from_local_directory[args1]
    This test runs pylint with `-j2` (parallel mode via multiprocessing forkserver) on a
    file that imports from a non-existent module (`completely_unknown`). The test expects
    pylint to suppress the import error. In both Python 3.14.3 and 3.14.5, the multiprocessing
    forkserver fails to start the pylint worker subprocess correctly (`BrokenPipeError: [Errno 32]
    Broken pipe`), causing a `CalledProcessError` and test failure. This is a pre-existing
    compatibility issue between pylint's parallel mode and Python 3.14's forkserver.
