  [Root cause classification]
  Two separate Docker-specific failures:

  (1) Docker host has 250 CPUs; cgroup v2 monkeypatch doesn't override actual cgroup files
    The three `test_cpu_count_cgroups_v2[2005-*]` tests check that dask's cpu_count function
    returns `min(host_cpus, cpu_quota/cpu_period)`. The test sets `quota=2005` and patches
    `/proc/self/cgroup` (via monkeypatch) to point to a group named `/`, `/user.slice`, or
    `/user.slice/more.slice`. Based on the quota, the expected CPU count is `ceil(2005/10) = 201`.
    The test expects `count == 201`, meaning the cgroup v2 quota should cap the CPU count.

    In the Docker environment, the `monkeypatch_cpu_count = None` fixture means the host's
    `os.cpu_count()` is not mocked and returns the actual host CPU count (250). The cgroup v2
    cpu quota limit calculation reads from `/sys/fs/cgroup/{group}/cpu.max`, but the
    monkeypatch only patches `/proc/self/cgroup` (the group discovery file), not the actual
    `/sys/fs/cgroup/cpu.max` quota file. The real container's cgroup file may have `max 100000`
    (no limit), so the quota calculation falls back to the host CPU count of 250.
    The assertion `assert 250 == 201` fails because the test's expected quota limit is not
    applied in the Docker container's cgroup hierarchy.

  (2) Docker root user bypasses file permission restrictions → config permission tests fail
    `test_collect_yaml_permission_errors[directory]` and `[file]` test that dask's config
    collector ignores YAML files and directories that cannot be read (permission denied).
    The tests use a context manager `no_read_permissions(path)` that calls `chmod(path, 0o000)`,
    then expects `dask.config.collect(...)` to return an empty dict (no config from the
    inaccessible file) or skip the inaccessible file in the returned config.

    Docker runs as root (uid=0). On Linux, root bypasses file permission checks — `chmod 000`
    does NOT prevent root from reading files. The dask config collector successfully reads the
    "restricted" YAML file because root has unrestricted access, producing `{'x': 1, 'y': 3,
    'z': 4}` instead of `{}`. The test expects a PermissionError to be silently caught, but
    no PermissionError occurs when running as root.
