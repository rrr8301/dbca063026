  [Root cause classification]
  - pyarrow 24.0.0 changed string dtype handling → test_parquet_read_pandas fails
    The test creates an Astropy `Table` with columns of various dtypes including `'U3'`
    (fixed-length NumPy unicode string, 3 chars). It converts the table to a pandas DataFrame,
    writes it as a parquet file (`df.to_parquet(version="2.4")`), then reads it back via
    `Table.read()`. The assertion `t2[str(dtype)].dtype == dtype` checks that the roundtrip
    preserves the dtype.

    In pyarrow 24.0.0 (installed in the Docker run), string columns from parquet files are
    now read back as ArrowDtype (large_string/object) instead of converting to NumPy fixed-length
    unicode strings. The dtype that was 'U3' (numpy.dtype('<U3')) is returned as 'object'
    (numpy.dtype('O')). The test expects `dtype('O') == 'U3'` which fails. In the original CI,
    an older pyarrow version (< 24.0.0) preserved the 'U3' dtype, so the test passed.

  - pytest.PytestUnraisableExceptionWarning for unclosed FileIO → test_models_fitting fails
    The `test_models_fitting[LevMarLSQFitter-model32]` failure is a teardown-phase unraisable
    exception: `Exception ignored in: <_io.FileIO name='/tmp/fits-test-.../data.txt' closefd=True>`.
    This means a FITS file handle opened during the test was not explicitly closed. Python's
    garbage collector finalizes it after the test, raising an unclosed resource warning. pytest
    9.x captures unraisable exceptions during teardown and converts them to test failures. The
    original CI's pytest version (or Python version) may not capture this warning, or the
    specific file handle lifecycle differs in that run.

  - The tests that fail in original but pass in LLM (test_new_fitsrec, test_appending_a_column)
    are FITS record manipulation tests that produce AstropyUserWarning about NULL values — these
    are environment-dependent and may relate to numpy or FITS library version differences.
