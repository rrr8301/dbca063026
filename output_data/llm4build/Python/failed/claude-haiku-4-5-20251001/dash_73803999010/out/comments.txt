  [Root cause classification]
  Two static type checker version bumps cause additional failures:

  (1) pyright 1.1.398 stricter type checking → 2 new pyright test failures
    The `test_typi002_typing_compliance` tests run pyright on sample Dash code to verify
    type safety. pyright 1.1.398 (installed from dash's pinned requirements) is stricter than
    the version used in the original CI. The sample test code uses `dash.Div(...)`,
    `html.H2(...)`, and `dcc.Input(value='', id='in')` — attributes that pyright 1.1.398
    reports as not known attributes of the `dash` module (`reportAttributeAccessIssue`).
    Additionally, pyright 1.1.398 detects missing required parameters in `Input()` call
    (`reportCallIssue`). The original CI's older pyright version accepted this code.
    NOTE: pyright 1.1.398 itself warns `there is a new pyright version available (v1.1.409)`
    indicating the installed version is not the latest; if the original CI used a newer version
    before 1.1.398 froze these checks, those tests would have passed.

  (2) mypy 1.15.0 requires specific `# type: ignore` error codes → 1 new mypy test failure
    mypy 1.15.0 enforces that `# type: ignore` comments must cover the actual error codes
    found at that line. In `dash/dash.py` lines 148-153, the code accesses `dash.html.Div`,
    `dash.dcc.Location`, etc. via dynamic imports. These lines have `# type: ignore` comments
    to suppress mypy errors, but the `# type: ignore` comment doesn't specify `[attr-defined]`.
    mypy 1.15.0 reports:
      `error: Module has no attribute "Div" [attr-defined]`
      `note: Error code "attr-defined" not covered by "type: ignore" comment`
    This means the `# type: ignore` is too generic (it covers `import` errors, not `attr-defined`
    errors). Additionally, mypy 1.15.0 finds a new error in `result_plotly_figure.py` about
    `str | None` vs `str` argument type mismatch for `PropRole.matches()`. None of these were
    flagged by the older mypy version in the original CI.
