  [Root cause classification]
  - litellm 1.82.6 changed the mapping behavior for unknown finish_reason values
    The test `test_finish_reason_unknown_maps_to_other` creates a mock LLM response with
    `finish_reason="eos"` — an unmapped value not in litellm's standard set (stop, length,
    tool_calls, content_filter). The adk-python codebase maps litellm finish_reason values
    to its own `types.FinishReason` enum and expects that any unknown/unmapped finish_reason
    value results in `FinishReason.OTHER`.

    In litellm 1.82.6 (installed in the Docker run), the behavior changed: unknown finish_reason
    values are no longer passed through as-is for the caller to handle. Instead litellm itself
    normalizes unknown values to a default known value — specifically `'stop'`. When adk-python's
    model adapter receives `finish_reason='stop'` from litellm, it maps it to `FinishReason.STOP`.
    The test expects `FinishReason.OTHER` but gets `FinishReason.STOP`.

    The original CI (2026-03-05) ran with a version of litellm where unknown finish reasons
    preserved the original string and were passed through differently, allowing adk-python to
    detect the unmapped value and return `FinishReason.OTHER`. The litellm version in the original
    CI was likely earlier than 1.82.6 where this normalization was not yet applied.

    The difference in which test failed is unrelated to Docker:
    - Original CI failure: `test_parallel_function_execution_timing` — a timing-sensitive flaky
      test measuring parallel function execution latency that failed under CI's specific load
      conditions; it passes in Docker under lower concurrency.
    - LLM failure: `test_finish_reason_unknown_maps_to_other` — fails due to litellm 1.82.6
      version drift that changed how unmapped finish_reason strings are returned to callers.
