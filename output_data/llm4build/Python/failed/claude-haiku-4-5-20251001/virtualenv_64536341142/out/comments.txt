  [Root cause classification]
  `pwsh` (PowerShell Core) not installed in Docker container → 2 new test failures

  virtualenv's PowerShell activation test (`test_powershell`) activates a virtual environment
  using PowerShell and runs commands in it. On Linux, the test uses `pwsh` (PowerShell Core)
  as the shell executable. The test defines:
    `cmd = "powershell.exe" if sys.platform == "win32" else "pwsh"`

  On GitHub Actions ubuntu-24.04 runners, PowerShell Core (`pwsh`) is pre-installed as part
  of the default tool suite. The test can discover the `pwsh` binary, create a PowerShell-
  activated virtualenv, and verify that activation works correctly.

  In the `ubuntu:24.04` Docker image, only the base system packages are present. PowerShell
  Core is not installed. When the test tries to execute `['pwsh', '-c', '$PSVersionTable']`
  to detect the PowerShell version, the subprocess raises:
    `FileNotFoundError: [Errno 2] No such file or directory: 'pwsh'`

  This propagates as a test failure rather than a skip — the test does not have a
  `@pytest.mark.skipif` conditional for missing `pwsh`, so the version detection failure
  causes the entire `activation_tester` to fail.

  The 2 additional skips in LLM (29 vs 27) are likely from other PowerShell-related tests
  that DO check for `pwsh` availability before running and skip gracefully when it's absent,
  where the original CI ran them (because `pwsh` was present).

  (2) Pre-existing failure (same in both): test_shim_uses_global_version_file
    This test patches `virtualenv.discovery.builtin.PathPythonInfo.from_exe` with a MagicMock
    and then calls virtualenv's Python discovery with a `.python-version` file pointing to
    `3.10.1`. The test expects `from_exe` to be called once. In the current version of
    virtualenv being tested, the shim resolution code path does not call `from_exe` when
    processing the global version file — possibly because the logic has been refactored to
    use a different internal path:
      `AssertionError: Expected 'from_exe' to have been called once. Called 0 times.`
    This test fails identically in both the original CI and LLM Docker, confirming it is
    a pre-existing issue with the code under test, not an environment difference.
