  [Root cause classification]
  SQLite database lock contention during concurrent bulk delete → 503 Service Unavailable
  → flaky test fails in original CI but passes in LLM Docker

  `test_bulk_delete_by_state_type` sends a bulk delete request to the Prefect orchestration
  API server that is backed by SQLite. The test runs under `--numprocesses=auto` with many
  parallel workers (`--dist=loadfile`). During concurrent test execution, multiple workers
  simultaneously perform database write operations. SQLite's single-writer lock model causes
  `sqlite3.OperationalError: database is locked` when multiple connections attempt to write
  to the same database simultaneously.

  The SQLite lock triggers the Prefect API server's error handler, which returns:
    `HTTP 503 Service Unavailable`
  The test expects HTTP 200 OK, so it fails: `assert 503 == 200`.

  In the LLM Docker, the test passes because the Docker environment has different timing
  and resource contention characteristics: with fewer background processes and a quieter
  system, the SQLite lock window is not triggered during test execution. The test's outcome
  depends on timing and concurrency, making it a **known flaky test**.

  Evidence from the git history: the repository has numerous active branches specifically
  aimed at fixing this flakiness:
    - `devin/1770145455-fix-flaky-bulk-delete-test`
    - `devin/1770390648-fix-flaky-bulk-delete-state-type`
    - `devin/1772646120-fix-flaky-bulk-delete-test`
    - `devin/1772663207-fix-flaky-bulk-delete-no-filter`
  These confirm the Prefect team has been actively investigating this flaky test.
