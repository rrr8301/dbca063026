  [Root cause classification]
  - setuptools-scm renamed version scheme `release-branch-semver` → DeprecationWarning treated
    as error by matplotlib's pytest filterwarnings configuration → 1051 test failures

    matplotlib's editable install generates its version string dynamically via
    `setuptools_scm.get_version()` (called as `mpl.__version__` property). The `pyproject.toml`
    specifies `version_scheme = "release-branch-semver"`. In the version of `vcs_versioning`
    (the internal engine used by newer setuptools-scm) installed in the Docker container, the
    scheme `release-branch-semver` was renamed to `semver-pep440-release-branch`. The old name
    is still supported but emits a `DeprecationWarning`.

    Any test that accesses `mpl.__version__` — either directly or indirectly through calling
    `fig.savefig(format="ps")`, `fig.savefig(format="pdf")`, `fig.savefig(format="svg")`, or
    other rendering code that embeds the version in the output file header — triggers this
    warning. matplotlib's `pyproject.toml` configures pytest with:
      `filterwarnings = ["error::DeprecationWarning"]`
    This converts the `DeprecationWarning` into an error, causing those tests to FAIL.

    1051 tests fail this way — all rendering tests that write to PS/PDF/SVG backends or otherwise
    access the matplotlib version. Tests using only the Agg backend (which doesn't embed the
    version string) pass successfully (2283 pass).

    The original CI uses a pinned/older setuptools-scm version (likely from pip's caching of the
    locked environment) where `release-branch-semver` is the correct current name, not yet
    renamed, so no DeprecationWarning is emitted and only 4 tests fail (pre-existing GTK4 issues).

  - --maxfail=50 stops test distribution early → only 3648 tests run vs 10250
    Since 1051 tests fail and pytest's `--maxfail=50` is set, pytest signals xdist workers to
    stop distributing new tests after 50 failures. However, with `-n auto` (many parallel workers),
    tests already queued in workers run to completion. The combination of many early failures +
    maxfail=50 stops pytest after only 3648 tests run (36% of the 10250 in original CI).

  - Additional teardown failures: `AttributeError: 'PosixPath' object has no attribute '_str'`
    pytest captures unraisable exceptions during teardown via its `unraisableexception` plugin.
    ~31 teardown errors occur where matplotlib's cleanup code (figure deletion in __del__) tries
    to compute a string representation of a PosixPath object that was created via
    pickle/unpickle across xdist workers. Python 3.12.3's pathlib.py caches `_str` lazily in
    `__str__` (line 441: `return self._str`), but pickled/reconstructed PosixPath objects
    lose this cached value. These teardown errors are separate from the main 1051 failures.
