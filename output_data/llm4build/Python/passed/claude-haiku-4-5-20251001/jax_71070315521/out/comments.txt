  [Root cause classification]
  - Python 3.11.0 bug in traceback._get_code_position triggers RuntimeError
    The Docker container uses `ubuntu:22.04` which ships Python 3.11.0 (the first
    public release). The original CI uses a GitHub Actions ubuntu-22.04 runner with
    a more recent Python 3.11.x (3.11.5+) installed via actions/setup-python.
    `testTracebackFromFrames` tests JAX's traceback manipulation by calling
    `traceback.format_tb(tb)` on a JAX-generated traceback. This invokes the
    CPython `_walk_tb_with_full_positions()` path, which calls `_get_code_position()`
    (`traceback.py:367`):
      `return next(itertools.islice(positions_gen, instruction_index // 2, None))`
    In Python 3.11.0, when `instruction_index // 2` exceeds the length of
    `positions_gen`, `next()` raises a bare `StopIteration`. Because this happens
    inside a generator function (`_extract_from_extended_frame_gen`), PEP 479
    (Python 3.7+) converts the `StopIteration` into `RuntimeError: generator
    raised StopIteration`. This CPython bug was fixed in Python 3.11.1 by adding
    proper bounds checking / default value handling in `_get_code_position`.
    On the CI runner with Python 3.11.5+, the fix is present and the test passes.
