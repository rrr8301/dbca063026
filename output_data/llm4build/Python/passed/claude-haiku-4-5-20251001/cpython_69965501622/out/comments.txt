  [Root cause classification]
  - Docker network stack / glibc NSS: string service name resolution fails with EAI_SERVICE
    All three failing tests call into glibc's `getaddrinfo()` using a string service name
    ("http") in place of a numeric port number.
    `testGetaddrinfo` calls `socket.getaddrinfo(HOST, "http")` directly.
    `testGetServBy` calls `socket.getservbyname()` and re-raises any failure as OSError.
    `test_create_connection_service_name` calls asyncio's `loop.getaddrinfo(host, "http",
    type=SOCK_STREAM)` which delegates to `_socket.getaddrinfo`.
    In all three cases the call returns `[Errno -8] EAI_SERVICE: Servname not supported
    for ai_socktype`. This error means glibc could not map the string service name "http"
    to a port number for the requested socket type combination.
    On the GitHub Actions ubuntu-24.04 runner, the system's NSS (Name Service Switch)
    `/etc/nsswitch.conf` is configured to look up the `services` database from `/etc/services`
    files, which defines "http" as port 80/tcp. In the Docker container (minimal ubuntu:24.04),
    the NSS configuration or the glibc service database lookup behaves differently, causing
    `EAI_SERVICE` to be returned when resolving string service names in certain socket type
    contexts. The 7 extra skipped tests in the LLM run (26 vs 19) are the gdb-based tests
    (test.test_gdb.*) skipped because `gdb` is not installed in the Docker container.
