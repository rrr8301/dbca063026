    tests per CI job. The splitting strategy (`--algorithm=least_duration`) requires a pre-built
    `durations/Linux.json` file containing per-test timing data from prior CI runs. In the original

    contains the same 151 tests as expected. In the LLM run, no such pre-existing file is present
    at the start; the log shows `[pytest-split] Splitting tests with algorithm: least_duration`
    (creating it fresh), and at the end `[pytest-split] Stored test durations in durations/Linux.json`.
    Without historical timing data, the test splitter distributes tests differently, putting only
    102 tests into group 3 instead of 151 -- 49 fewer tests run.

  (2) PowerShell not installed -- test_shell_available[powershell] fails with FileNotFoundError
    `tests/shell/test_powershell.py::test_shell_available[powershell]` checks that a PowerShell
    interpreter (any of: `powershell`, `pwsh`, `pwsh-preview`) is accessible on PATH. In the Docker
    container (ubuntu:22.04), no PowerShell interpreter is installed -- the Dockerfile installs
    shell helpers (ash, csh, fish, tcsh, zsh) but not PowerShell. The test fixture raises:
    `FileNotFoundError: Shell(name=('powershell', 'pwsh', 'pwsh-preview'), path=None, exe=None) not found`
    The GitHub Actions runner (ubuntu-latest) has PowerShell pre-installed as `pwsh`, so the test
    passes in CI. This test happens to fall in group 3 of the LLM's split and fails at setup (ERROR),
    while the original's group 3 may have excluded it due to different split boundaries.
