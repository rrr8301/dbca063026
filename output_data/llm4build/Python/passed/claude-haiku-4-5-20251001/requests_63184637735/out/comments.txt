  - Missing test certificate file tests/certs/valid/ca/ca.crt
    The test `test_different_connection_pool_for_tls_settings_verify_bundle_unexpired_cert`
    (in `TestPreparingURLs`) tests that requests correctly differentiates connection pools
    based on TLS `verify` settings. It starts a local TLS server with:
      `TLSServer(cert_chain="tests/certs/valid/server/server.pem",
                 keyfile="tests/certs/valid/server/server.key")`
    and then makes a request with a specific CA bundle:
      `r2 = s.get(url, verify="tests/certs/valid/ca/ca.crt")`
    The server starts successfully (server.pem and server.key exist), but
    `tests/certs/valid/ca/ca.crt` is not found in the Docker container. The requests
    library checks `os.path.exists(cert_loc)` in `adapters.py:305` and raises
    `OSError: Could not find a suitable TLS CA certificate bundle`.
    This test is a new addition for Python 3.14 testing, testing TLS CA bundle validation.
    The test certificate hierarchy (server cert + CA cert) appears to require a generation
    step that produces `tests/certs/valid/ca/ca.crt`. On the GitHub Actions runner
    (Python 3.14 CI), this file exists either because it's committed to the repo or because
    a CI setup step (not captured in `make` or `make ci`) generates it using `trustme`
    (which is in requirements-dev.txt). In the Docker container, `make` only installs
    dependencies without generating the certificate files, leaving `ca.crt` absent.
    Note: both `server.pem` and `server.key` do exist (server starts, and a first request
    `r1 = s.get(url, verify=False)` succeeds), while only `ca.crt` in the CA subdirectory
    is missing -- consistent with partial cert generation.
