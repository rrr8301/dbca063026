  [Root cause classification]
  Two distinct failure modes:

  (1) Docker root user bypasses filesystem permission checks -- test_collect_yaml_permission_errors
    (2 tests: test_collect_yaml_permission_errors[directory], test_collect_yaml_permission_errors[file])
    These tests verify that `dask.config.collect_yaml()` gracefully handles unreadable directories
    and files. Each test creates two YAML config files (a.yaml: {x:1,y:2}, b.yaml: {y:3,z:4}),
    then uses a `no_read_permissions(cant_read)` context manager to apply `chmod 000` to either
    the whole directory or just a.yaml. It then calls `collect_yaml(paths=[tmp_path])` and expects
    the config to omit the content from the unreadable path:
    - `[directory]` test: expects `collect_yaml()` to return `{}` (empty -- can't read the dir)
    - `[file]` test: expects `collect_yaml()` to return `{y:3, z:4}` (only b.yaml readable)
    When running as root in Docker, `chmod 000` has no effect -- root always has read access.
    Both YAML files are read successfully, so `collect_yaml()` returns `{x:1, y:3, z:4}` (the
    merged content of both files), causing both assertions to fail.
    On the GitHub Actions runner (non-root user), `chmod 000` actually restricts read access.
    The test path also confirms it: `/tmp/pytest-of-root/pytest-0/popen-gw2/...` shows user=root.

  (2) Different cgroup v2 CPU quota configuration -- test_cpu_count_cgroups_v2 fails
    (3 tests: quota=2005, group_name in [/, /user.slice, /user.slice/more.slice])
    These tests verify `dask.system.cpu_count()` correctly interprets cgroup v2 CPU quota limits.
    They monkeypatch `builtins.open` to inject fake cgroup files:
      /proc/self/cgroup → "0::{group_name}"
      /sys/fs/cgroup/{group_name}/cpu.max → "2005 10"
    With quota=2005 and period=10, expected: ceil(2005/10) = 201 CPUs.
    But the tests get `assert 250 == 201` -- `cpu_count()` returns 250 (the actual number of
    logical CPU cores on the Docker host machine). This happens because the Docker container
    runs on a host with 250 CPUs. The `monkeypatch_cpu_count=None` parameter means the
    `multiprocessing.cpu_count()` fallback is NOT mocked. When the cgroup quota parsing code
    fails to intercept the patched open() calls (e.g., in a subprocess spawned by the pytest
    worker or when cgroup detection takes a different code path), it falls back to the real
    `multiprocessing.cpu_count()` which returns 250. On the GitHub Actions runner with fewer
    physical CPUs (where cpu_count() returns a smaller number), the cgroup quota of 201 is
    binding and the test passes.
