  [Root cause classification]
  Dependency version conflict between two incompatible versions of `vergen_lib` in the
  transitive dependency graph causes compilation of `kcl-version`'s build script to fail.

  The `crates/version/build.rs` uses `vergen_gitcl::*` which brings in `vergen-gitcl`
  as a dependency. This in turn depends on `vergen_lib-0.1.6` for its `Add` trait. At
  the same time, another dependency pulls in `vergen-9.1.0` which uses `vergen_lib-9.1.0`
  and its own definition of the `Add` trait. Rust treats these as distinct types from
  distinct crates (even with the same name), so `vergen::feature::rustc::Rustc` (which
  implements `vergen_lib-9.1.0::Add`) does not satisfy the required `vergen_lib-0.1.6::Add`
  bound in `build.rs:10`.

  The original CI ran successfully because it used a pinned `Cargo.lock` that resolved
  all `vergen_lib` dependents to the same version. The LLM Docker either doesn't use
  `--locked` to enforce the Cargo.lock, uses a different Rust toolchain version that
  causes Cargo to re-resolve dependencies, or the LLM's generated Dockerfile updates
  dependencies before building, pulling in a newer version of `vergen` or `vergen_gitcl`
  that introduces the version split.

  After 20 attempts, none of the LLM-generated Dockerfiles and run scripts could work
  around the fundamental compilation failure. The kcl-version build script fails on every
  attempt, preventing any test execution.
