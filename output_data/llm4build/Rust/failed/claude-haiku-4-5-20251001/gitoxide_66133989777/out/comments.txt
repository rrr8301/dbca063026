  [Root cause classification]
  gitoxide's test suite uses a hash-based fixture system where test fixtures are either
  extracted from pre-built archives (`tests/fixtures/generated-archives/*.tar`) or
  regenerated by running shell scripts. The fixture creation hash ("Using hash 'sha1'
  when determining which fixture to use or recreate") is computed from the fixture content,
  not from a platform hash — so fixtures can diverge when the generating scripts produce
  different git object IDs across architectures.

  Multiple failure categories:

  (1) Clone test: "not found on disk" (both original CI and LLM)
    Tests like `clone::blocking_io::fetch_and_checkout` create a local git repository,
    spawn it as a git server, clone it, check out a working copy, and then assert that
    specific files exist on disk at a temp path. Both the aarch64 CI and x86_64 Docker
    fail with `"/tmp/.tmpXXX/file" not found on disk` at `gix/tests/gix/clone.rs:688`.
    This is a pre-existing fixture regression on both platforms — the expected file at
    the tmp clone destination isn't being checked out correctly.

  (2) Tree diff object type mismatch: "left: Commit right: Blob" (both environments)
    `object::tree::diff::changes_against_tree_modified` expects to navigate from a tree
    entry to a Blob, but receives a Commit. This fixture produces different object types
    than expected, indicating stale fixture data committed to the repo no longer matches
    the current codebase behavior. Both aarch64 and x86_64 hit this.

  (3) Architecture difference (9 additional LLM failures vs original)
    The original CI runs on ARM64 (aarch64-unknown-linux-gnu) with `GIX_TEST_CREATE_ARCHIVES_EVEN_ON_CI=1`,
    which regenerates fixture archives during the CI run. The LLM runs on x86_64 without
    this env var, relying on committed archives. Architecture-specific differences in
    fixture generation scripts (e.g., git pack file format differences, path handling)
    cause 9 additional tests to fail in the LLM.

  (4) Missing jq utility (LLM Docker only — 2 failures)
    Tests in `gix-path` that probe git's environment use `jq` to parse JSON output.
    The Docker image doesn't have `jq` installed:
      `jq utility is available in PATH: Os { code: 2, kind: NotFound, message: "No such file or directory" }`

  (5) Terminal/PTY interaction (LLM Docker only — 2 failures with LEAK)
    `gix-prompt::prompt ask::username_password` and `ask::askpass_only` interact with
    PTY/terminal for password prompting. These tests hang and are killed after 21-22s,
    leaking resources. Docker containers without a TTY allocated cannot properly simulate
    terminal interactions for credential prompting.
