  [Root cause classification]
  Both tests belong to fuel-core's consensus module Proof-of-Authority (PoA) leader lock
  implementation that uses Redis for distributed state. These tests simulate multi-node
  scenarios — acquiring Redis leases, competing for leadership, epoch tracking, and block
  reconciliation — all using real Redis timing and key expiry (PEXPIRE).

  The tests are inherently non-deterministic: they rely on timing windows for Redis TTL
  operations and tokio async task scheduling. Two different timing-sensitive assertions fail:

  (1) Original CI: `has_lease_owner_quorum__adopts_higher_epoch_from_expanded_node`
    The test expands a lock to a new node C and expects the leader to advance to epoch 2
    before node C can acquire at epoch 1. The assertion is:
      `Leader's epoch (1) should be > node C's pre-acquisition epoch (1)`
    The epoch counter stayed at 1 instead of advancing to 2 — a race condition where the
    lock expansion and epoch bump didn't complete before node C read its epoch.

  (2) LLM Docker: `unreconciled_blocks__when_redis_node_restarts_and_loses_data__drops_block_below_quorum`
    The test simulates a Redis node restart (data loss) and expects the leader to detect
    the data loss and drop the affected block from the quorum set. Instead, the lock was
    lost to another simulated leader during the repair operation:
      `repair error: Lost lock during repair — another leader took over`
    The simulated lock loss occurred faster than expected due to Docker timing differences
    vs the original CI environment.

  Both are pre-existing flaky tests that test the same Redis leader-lock consensus logic
  from different angles. The failure manifests non-deterministically depending on scheduling
  latency, Redis operation timing, and system load. On the LLM's 4th attempt it converged
  on a different flaky test failing than on the original CI run.
