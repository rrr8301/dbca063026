  [Root cause classification]
  Two separate failure causes:

  (1) LLM only: OOM (code 12) in disk::fs and disk::local tests
    All 34 LLM failures are in the `rustfs-ecstore` crate's disk layer tests
    (`crates/ecstore/src/disk/fs.rs` and `crates/ecstore/src/disk/local.rs`). These tests
    exercise file system operations: open, read, write, rename, stat, remove, and cleanup.
    Every test panics with `Os { code: 12, kind: OutOfMemory, message: "Cannot allocate memory" }`.

    The `rustfs` project uses io_uring-style async disk operations that require the kernel
    to pin pages in locked memory (RLIMIT_MEMLOCK). Docker containers without `--privileged`
    or `--ulimit memlock=-1` have a very low default locked memory limit (typically 64KB).
    When the disk layer tries to allocate I/O ring buffers or large file-backed buffers for
    its async operations, the kernel returns ENOMEM (error 12). The unwrap at each file
    operation site panics when the allocation fails.

    On GitHub Actions, the runner has no RLIMIT_MEMLOCK restriction, so all disk operations
    allocate their buffers normally. In the Docker container, every disk I/O operation that
    requires locked memory fails with ENOMEM.

  (2) Original CI only: Scanner lifecycle test fails with metadata read error
    The `test_scanner_expires_zero_day_current_version` test validates that the rustfs
    lifecycle scanner correctly expires objects with zero-day lifecycle rules on the current
    version. The test panics at `lifecycle_integration_test.rs:571` with:
      `scanner get_size should succeed: Io(Custom { kind: Other, error: "failed to read metadata" })`
    This is a pre-existing lifecycle integration test failure where the scanner's `get_size`
    call cannot read object metadata. The LLM doesn't fail this test — likely because the
    full lifecycle integration test suite requires a running rustfs server instance, and the
    LLM Docker doesn't start the server.

  (3) Test count difference: original=4042 vs LLM=1913 (2129 fewer in LLM)
    The original CI runs the full workspace including `rustfs-scanner` lifecycle integration
    tests (4042 total). The LLM runs a narrower scope (1913 tests) — the lifecycle scanner
    tests are absent because the LLM's run.sh doesn't configure the necessary service
    dependencies (embedded server, S3 endpoints) for the integration tests to run.
