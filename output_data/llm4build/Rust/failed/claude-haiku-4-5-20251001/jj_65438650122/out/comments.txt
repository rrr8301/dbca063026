  [Root cause classification]
  Three separate root causes explain the 7 additional LLM failures:

  (1) Pre-existing: 3 insta snapshot failures (both original CI and LLM)
    The `test_workspaces` tests use the `insta` crate for snapshot testing. Snapshots
    stored in the repository under `cli/tests/snapshots/` no longer match the current
    workspace command output. Both environments fail because the code behavior changed
    but committed snapshots weren't updated. The assertion:
      `snapshot assertion for 'workspaces_rename_workspace_from_before_workspace_store-2' failed`
    indicates the actual output diverges from the stored baseline.

  (2) File lock timeout in Docker (LLM only — 1 failure)
    `lock::tests::lock_concurrent::fallback_filelock_lock_expects` tests concurrent file
    locking by spawning threads that all try to acquire the same lock. On Docker with higher
    I/O latency or different scheduler behavior, one thread holds the lock file (`code: 17,
    AlreadyExists`) longer than the timeout allows, causing:
      `FileLockError { message: "Timed out while trying to create lock file" }`
    On GitHub Actions with faster disk I/O, the lock acquisition completes before timeout.

  (3) Root user bypasses file permissions (LLM only — 1 failure)
    `test_local_working_copy::test_check_out_existing_file_cannot_be_removed` tests that
    jj cannot check out a file if an existing path element can't be removed (e.g., a
    read-only directory). In Docker running as root, `chmod 000` on a directory doesn't
    prevent root from removing it — root ignores Unix file permission checks. On GitHub
    Actions (non-root user), the permission restriction is enforced. The test fails
    because the "cannot be removed" scenario actually succeeds when run as root.

  (4) OpenSSH signing tools not installed in Docker (LLM only — 5 failures)
    Five `test_ssh_signing` tests call `ssh-keygen -Y sign` and `ssh-keygen -Y verify-blob`
    to create and verify SSH-signed commits. The Docker image doesn't have OpenSSH installed
    or lacks the signing-capable version of `ssh-keygen`. Every ssh_signing test that
    invokes the `ssh-keygen` subprocess gets:
      `Backend(Io(Os { code: 2, kind: NotFound, message: "No such file or directory" }))`
    On GitHub Actions, OpenSSH is pre-installed with full signing support.
