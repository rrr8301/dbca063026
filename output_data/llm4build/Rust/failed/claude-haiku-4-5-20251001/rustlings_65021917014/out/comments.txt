  [Root cause classification]
  Two separate failure causes across platforms:

  (1) LLM only (Linux): clippy component missing from Rust toolchain
    The LLM Dockerfile uses `FROM rust:latest` without installing the clippy component.
    When `rustlings init` is called in the `init` integration test, rustlings checks whether
    clippy is installed before initializing. The check at `src/init.rs:55` calls rustup to
    verify clippy is available and fails with "Clippy, the official Rust linter, is missing.
    Please install it first before initializing Rustlings."

    Similarly, `run_compilation_success` and `run_test_success` tests invoke rustlings to
    compile/test an exercise file, which internally runs `cargo-clippy`. These fail with:
    `error: 'cargo-clippy' is not installed for the toolchain '1.95.0-x86_64-unknown-linux-gnu'`

    The fix: `rustup component add clippy` in the Dockerfile. The official `rust:latest`
    Docker image ships the Rust toolchain without optional components like clippy, which
    must be explicitly installed.

  (2) Original CI (Windows): init fails with Windows-specific path issue
    On Windows, the `init` test fails at `tests\integration_tests.rs:170`. The Windows
    init path (`src/init.rs:123`) fails for a different reason — likely a Windows-specific
    issue with the rustlings initialization directory structure, git setup, or file system
    path handling. The LLM doesn't replicate this Windows-specific failure because it runs
    on Linux.

  (3) Platform difference (Windows vs Linux): different test behaviors
    The rustlings test suite is designed to run on Windows (CI environment). On Linux Docker,
    the tests encounter a different set of failures due to the missing clippy component.
    The 2 additional LLM failures (`run_compilation_success`, `run_test_success`) are
    Linux-Docker-specific and don't appear on Windows because clippy is pre-installed in
    the GitHub Actions Windows runner.
