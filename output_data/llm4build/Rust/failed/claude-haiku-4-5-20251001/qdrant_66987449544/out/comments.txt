  [Root cause classification]
  Multiple Docker environment limitations cause different failures than the original CI:

  (1) io_uring not supported in Docker (RLIMIT_MEMLOCK restriction)
    io_uring requires the kernel to allocate locked (non-swappable) memory pages for its
    submission queue ring buffers. Docker containers without `--privileged` or the
    `CAP_IPC_LOCK` capability have a very low default `RLIMIT_MEMLOCK` (typically 64KB),
    which is insufficient to initialize io_uring's internal ring buffers. The qdrant
    `universal_io::io_uring` module calls `io_uring_setup()` which fails with:
      `IoUringNotSupported("failed to initialize io_uring: Cannot allocate memory (os error 12)")`
    io_uring requires either `--privileged` or `--ulimit memlock=-1` docker flags.
    On GitHub Actions, the runner has no such restriction and io_uring initializes normally.

  (2) Memory-mapped vector operations fail with OOM in Docker
    Qdrant's `async_raw_scorer` tests use memory-mapped files (`mmap`) to score large
    vector datasets asynchronously. Creating large mmap regions (needed for scoring
    thousands of vectors) fails in Docker with:
      `OutOfMemory { description: "IO Error: Cannot allocate memory (os error 12)", free: 0 }`
    Docker containers have constrained virtual memory settings. When the kernel's OOM
    killer or mmap restrictions prevent large anonymous mappings, qdrant's async IO
    layer (which uses io_uring for async mmap reads) fails to allocate the needed memory.
    This also cascades to the `custom_query_scorer_equivalency` tests that use
    `async_memmap_storage` as the backend.

  (3) Flush topology circular dependency bug (intermittent)
    `continuous_snapshot_test::test_continuous_snapshot` fails due to a pre-existing race
    condition in qdrant's segment flush topology:
      `circular dependencies detected in flush topology: [171, 173]`
    This indicates that two segment flush tasks have been scheduled with a dependency
    cycle. This is a timing-sensitive bug that surfaces under Docker's slower I/O
    and CPU scheduling — concurrent snapshot creation and segment flushing creates
    a circular wait in the flush dependency graph.

  (4) Test scope difference: 15734 (LLM) vs 1527 (original CI)
    The original CI uses `--profile ci --locked` which applies a custom nextest profile
    from `.cargo/nextest.toml` that limits test scope (e.g., excludes slow integration
    tests, uses a max-fail threshold). The LLM runs `cargo nextest run --workspace`
    without `--profile ci`, executing the full test suite including all integration and
    performance tests. This explains the 10x difference in test counts.
