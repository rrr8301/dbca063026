  [Root cause classification]
  Two separate failure causes:

  (1) Pre-existing (both original CI and LLM): insta snapshot mismatch in return_type.md
    The `ty_python_semantic` mdtest for `function/return_type.md` tests generator function
    return type inference. The committed snapshot at
    `crates/ty_python_semantic/resources/mdtest/snapshots/return_type.md_-_..._Synchronous_....snap`
    reflects the old behavior, but the current implementation now also emits
    `error[invalid-assignment]: Object of type `Literal[42]` is not assignable to `None``
    for generator functions returning None-typed yields. The new output contains additional
    diagnostic lines (at positions 37, 67, 70 in the test file) not in the old snapshot.
    Both original CI and LLM hit this identical snapshot assertion failure.

  (2) LLM only: `ruff::integration_test unreadable_pyproject_toml` passes unexpectedly
    This integration test creates a temporary directory with a `pyproject.toml` whose
    permissions are set to 000 (unreadable by non-root users). It then runs ruff against
    that directory, expecting ruff to fail with an error because it cannot read the config
    file. In Docker, the container runs as root. Root bypasses Unix file permission checks
    and can read any file regardless of its permission bits. So ruff successfully reads the
    `pyproject.toml`, runs without error, and exits with status 0. The test expected a
    non-zero exit but got 0 → `Error: Unexpected success`. On GitHub Actions, the runner
    is a non-root user, the chmod 000 file is genuinely unreadable, and ruff fails correctly.
