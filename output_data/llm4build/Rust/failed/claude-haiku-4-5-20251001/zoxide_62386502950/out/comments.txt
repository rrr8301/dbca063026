  [Root cause classification]
  Two distinct failure patterns with different root causes:

  (1) Original CI only: xonsh_black Black formatter code style failure
    The `shell::tests::xonsh_black` tests generate zoxide's xonsh shell integration script
    and validate it against Black, the Python code formatter. The test runs:
      `black --check --diff -` on the generated xonsh script.
    Black reports "1 file would be reformatted" — the generated xonsh script doesn't comply
    with Black's formatting rules. This is a pre-existing code style regression: the xonsh
    script generator produces code that Black would reformat. All 24 failing tests are
    xonsh_black test parameterizations (different combinations of cmd/hook/echo/resolve_symlinks).
    The original CI's nix-shell provides `black` (Python formatter) and `xonsh`, so this
    test reaches the formatting stage and fails.

  (2) LLM only: all shell tool tests fail due to missing system tools
    The LLM Docker doesn't install any of the shell tools required by zoxide's integration
    tests. The 387 LLM failures span:
    - `completions_powershell`: `pwsh` (PowerShell) not installed → "No such file or directory"
    - `completions_fish`: `fish` (Fish shell) not installed → "No such file or directory"
    - `completions_zsh`: zsh integration test fails (shell-specific test)
    - `bash_shellcheck`: `shellcheck` (bash linter) not installed → "No such file or directory"
    - `bash_shfmt`: `shfmt` (bash formatter) not installed → "No such file or directory"
    - `xonsh_black`: `black` (Python formatter) not installed → "No such file or directory"
    - Various other shell-specific integration tests

    The original CI uses `nix-shell --pure` which creates an isolated environment with
    precisely the tools declared in the nix derivation (black, shellcheck, shfmt, fish,
    xonsh, pwsh via wine, etc.). The Docker container has none of these, so every test
    that invokes an external shell tool fails immediately with ENOENT.

  (3) Cargo binary count difference: original=11952, LLM=191244
    The massive cargo binary count difference (11952 vs 191244) reflects that cargo's
    test counting includes all test permutations in the parameterized test matrix.
    The parameterized shell tests generate many test binaries when run with the full
    feature set on a system where the underlying shell tools are present/absent differently.
