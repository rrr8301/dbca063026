  [Root cause classification]
  - Two distinct Docker environment issues cause the task count mismatch:
    (1) Android SDK not available in Docker. The NullAway project includes a `sample-app`
    subproject that requires the Android SDK (`ANDROID_HOME` env var or `local.properties`
    with `sdk.dir`). Docker has no Android SDK installed, so `./gradlew build` fails at
    `:sample-app:compileDebugJavaWithJavac` after only 7 tasks. Similarly,
    `./gradlew codeCoverageReport` fails for the same reason
    (`:jar-infer:test-android-lib-jarinfer:extractReleaseAnnotations` cannot resolve
    dependencies without the Android SDK). In the original CI (GitHub Actions with
    `setup-gradle@v5` which configures the Android build environment), all tasks execute
    and cache successfully.
    (2) `./gradlew shellcheck` fails in Docker because the LLM-generated run.sh declares
    `COVERAGE_SUCCESS=false` but never exports or uses it after the if-block, triggering
    shellcheck SC2034 ("appears unused"). The original CI run.sh does not have this unused
    variable.
    The net effect: `./gradlew publishToMavenLocal` (the only step that succeeds fully)
    runs without the pre-built artifact cache from `./gradlew build`, so it must compile
    all modules from scratch — 78 tasks executed vs 38 in the original (where prior build
    steps had already cached compiled artifacts via `gradle/actions/setup-gradle`), which
    explains the task count mismatch that marks the run as non-fully-reproducible.
