  [Root cause classification]
  - Docker network stack immediately rejects connections to unreachable hosts → 8 connection
    timeout tests fail. The `TestAsyncConnectionTimeouts` and `TestConnectionTimeouts` tests
    configure HTTP connections with a short connect timeout (e.g., 100ms) and connect to a
    TCP address that should time out (typically a non-routable IP or a host that silently drops
    packets). The tests assert that `ConnectTimeoutException` is thrown after the timeout period.
    In Docker's network stack, when the kernel determines a destination IP is not reachable
    (ENETUNREACH — "Network is unreachable"), it immediately returns an error without waiting
    for the timeout, raising `java.net.SocketException: Network is unreachable` instead of
    the expected timeout. The test expects the slow "wait then timeout" behavior of
    `ConnectTimeoutException`, but Docker delivers the fast "immediate unreachable" behavior
    of `SocketException`, causing the exception type assertion to fail.
    In the original CI (GitHub Actions Ubuntu with full network routing), the unreachable
    address causes a timeout rather than an immediate rejection, so `ConnectTimeoutException`
    is correctly thrown and the tests pass.
    The test count mismatch (2683 vs 2559 = 124 fewer) is because the LLM uses system Maven
    (`mvn`) instead of `./mvnw`, and without the wrapper's pinned version, some tests in
    downstream modules may be skipped or not resolved correctly.
