  [Root cause classification]
  Three distinct failure modes:

  (1) Docker root user bypasses filesystem permission checks -- TestFileSingleStreamSpillerFactory
    (2 tests: testDistributesSpillOverPathsBadDisk, testCacheInvalidatedOnBadDisk)
    These tests simulate a "bad disk" by creating a temp directory and making it non-writable
    with `chmod 000` / `setReadable(false)`. They then expect the spiller either to avoid
    writing spill files to that path, or to throw an IOException when attempting to write.
    When the container runs as root, Unix permission bits (including mode 000) are ignored for
    root processes. So:
    - `testDistributesSpillOverPathsBadDisk` expects no spill files in the "bad" directory,
      but root writes there anyway -> `Expecting empty but was: [/tmp/.../spill...bin, ...]`
    - `testCacheInvalidatedOnBadDisk` expects an IOException from writing to the bad directory,
      but root succeeds silently -> `Expecting code to raise a throwable`
    The original CI runner is a non-root user where `chmod 000` actually restricts access.

  (2) No Docker socket -- Testcontainers-dependent OAuth2 tests fail
    (3 test classes: TestOAuth2WebUiAuthenticationFilterWithJwt, ...WithOpaque,
     ...WithRefreshTokens)
    The Hydra identity provider (an OAuth2/OIDC server used in tests) is started via
    Testcontainers (`TestingHydraIdentityProvider.start()`), which requires access to
    `/var/run/docker.sock`. The Docker container running the build does not expose the host
    Docker socket, so Testcontainers throws:
    `IllegalStateException: Could not find a valid Docker environment`.
    Once the first class fails to find Docker, subsequent classes immediately fail with
    `Previous attempts to find a Docker environment failed. Will not retry.`

  (3) Resource exhaustion in Docker container causes distributed query timeouts
    (22 tests across: TestBinPackingNodeAllocator, TestWebUi, TestRecursiveCte,
     TestConnectorTracingContextPropagation, TestJsonExtract, TestBigintOperators,
     TestVarbinaryFunctions, TestDriver)
    Trino's integration tests start an embedded single-node Trino coordinator+worker per
    test class, executing real SQL queries via HTTP. When many test classes run in parallel
    inside a resource-limited Docker container (shared CPU, memory), several problems arise:
    - `TestBinPackingNodeAllocator`: The allocator test asserts that node assignment
      completes within 2500ms. Under Docker resource contention, the scheduler thread
      cannot respond in time -> `TimeoutException: timed out after 2500 milliseconds`
    - `TestWebUi`: The embedded HTTPS server takes 56s-4874s to respond to HTTP requests
      -> `SocketTimeoutException: timeout` for all 7 tested auth flows
    - `TestRecursiveCte`: The embedded worker node becomes unresponsive for over 60s
      -> `PageTransportTimeoutException: 7 failures, failure duration 61.55s`. All 6 queries
      that involved the slow worker node fail; 6 tests complete after 10526 total seconds
    - `TestConnectorTracingContextPropagation`, `TestJsonExtract`, `TestBigintOperators`,
      `TestVarbinaryFunctions`, `TestDriver`: All exceed 10000+ seconds. The embedded Trino
      task management marks tasks as abandoned after 2 minutes of inactivity, and JWT tokens
      used for internal auth expire mid-test (JWT expired 2589971 ms ago).
    The original CI runs on a dedicated GitHub Actions runner with sufficient CPU/memory,
    completing all tests within 178 seconds wall time. The LLM run completes in 154 seconds
    total, but the parallel test execution causes some slow queries to time out.
