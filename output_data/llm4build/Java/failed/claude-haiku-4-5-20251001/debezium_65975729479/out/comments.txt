    does not exist, so all Testcontainers-based integration tests fail with
    "Could not find a valid Docker environment":
    The `debezium-testing-testcontainers` module contains integration tests that use
    Testcontainers to spin up real Docker containers for MongoDB replica sets, Apicurio Schema
    Registry, Debezium Connect clusters, and MySQL. Testcontainers requires access to the Docker
    daemon via `/var/run/docker.sock`. On the GitHub Actions runner, Docker is available natively
    (the runner has Docker installed and the socket is accessible). Inside the LLM's Docker
    container, no Docker-in-Docker (DinD) setup is configured — the Docker socket is not mounted
    into the container, and there is no Docker daemon running. Testcontainers detects this on first
    use (`MongoDbReplicaSetContainerIT.testCluster`) and throws `IllegalStateException: Could not
    find a valid Docker environment`. Subsequent tests that use Testcontainers (`ApicurioRegistryTestIT`,
    `DebeziumContainerIT`) get a cached "previous attempts failed" error without re-checking. The
    original CI's 1 failure (`MongoDbReplicaSetAuthContainerIT`) is a different Docker-related issue:
    Docker IS available there, but the MongoDB replica set setup fails with a MongoDB authentication
    error ("replSetGetStatus requires authentication") — the test works in the original CI up to the
    point of running MongoDB commands. The LLM also ran 7 fewer tests (860 vs 867), likely because
    the LLM's run.sh omits the `-Pmysql-ci` profile and other debezium-specific flags, causing some
    test classes to be compiled or filtered differently.
