    causing `JiraCsv196Test` to decode multi-byte UTF-8 sequences into wrong Unicode codepoints:
    `JiraCsv196Test` (JIRA issue CSV-196) tests that the CSV parser correctly reads records
    containing 3-byte and 4-byte UTF-8 encoded characters. It asserts specific Unicode codepoints
    at index 3 of the parsed result: 242 (U+00F2) for the 3-byte test, 701 (U+02BD) for the
    4-byte test. On the GitHub Actions runner, JDK 11's default charset is UTF-8 (the runner has
    `LANG=en_US.UTF-8`), so the CSV parser reads the byte stream as UTF-8 and decodes the
    multi-byte sequences to the expected codepoints. In the LLM's Docker container (ubuntu:22.04
    without `LANG` or `LC_ALL` set), the default charset is ANSI_X3.4-1968 (7-bit ASCII). With
    ASCII as the charset, Java's reader treats each byte of a multi-byte UTF-8 sequence individually
    as an ASCII/Latin-1 value. The high bytes of the multi-byte sequence (0x80-0xFF range) are
    mapped to their Latin-1 equivalents and combined into different codepoint values: index 3 comes
    back as 1901 and 398 respectively instead of 701 and 242. The same `-Drat.skip=true
    -Dlicense.skip=true` flag differences in the LLM command have no effect on this test.
